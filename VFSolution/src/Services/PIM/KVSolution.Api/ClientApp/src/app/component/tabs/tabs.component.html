 Cont

For the first time when I heard about “microservice”, I thought that there is a small service only serving one purpose in the application. Yup … that ‘s my firstly idea about “microservice”. And after more than two years with microservice architecture, from the basic to advance I see that it ‘s not simple like that. Each microservice will be alive single without any coupling in other domain service. From my experience, if we follow microservice we shouldn’t develop the “Core application” for the whole our application or something like that because of that mindset will depend on “Core”, it will make us breaking service more difficultly in the feature. Microservice can be built by any techniques, platform with any structure, layer … depend on the requirement, ability of team. Some new concept like serverless framework is available from Cloud service support very good for microservice architecture like AWS Lamda or Azure Function. Example we have to develop the payment solution. We can have payment service using .NET technology, order service using Node JS technology. Each service will stand alone and communicate each other by event message or make the request to other service directly. And if there is one service go down for any reason, the other still can be alive. With the modern style of deployment using cloud service, we can scale the service easily depend the demand or traffic. Okay told too much about the concept … let ‘s continue with approach.
